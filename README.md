# glossa

**Phoneme ↔ Grapheme Explorer** — browse the sound systems of the world's
languages, see IPA charts filtered to each language's inventory, explore how
graphemes map to phonemes, and hear each phoneme pronounced.

Frontend-only, static-deployable (Vite + React + TypeScript). Published at
**https://apps.thorwhalen.com/glossa/**.

> This project was previously called **phogra**. The old URL
> `https://apps.thorwhalen.com/phogra/...` is preserved as a redirect to
> `/glossa/...` (see [URL back-compat](#url-back-compat) below).

- [x] **M1** — Universal IPA chart + Commons audio

- [x] **M6** — Compare two languages
- [x] **M7** — Polish (dark mode, copy-IPA, font self-hosting, code splitting)

Redeploy with `./scripts/deploy.sh` (builds, rsyncs to
`/opt/tw_platform/apps/glossa/frontend/`, refreshes the `/phogra/`
redirect stub, sends `SIGHUP` to the gunicorn master for a zero-downtime
reload).

> HUP is the right signal here because only the static frontend changed.
> After a venv mutation (`pip install`, editable reinstall, upgrade) you
> must `systemctl restart enlace-backend` instead — HUP'd workers inherit
> the master's stale `sys.path` and can fail to boot with
> `ModuleNotFoundError`. See `.claude/CLAUDE.md` for the full rule.

## Running locally

```bash
# install JS deps
npm install

# regenerate the data bundle (see "Data pipeline" below)
cd data-prep && uv sync && uv run glossa-data-prep run-all && cd ..

# dev server
npm run dev

# production build
npm run build
```

Open http://localhost:5173, click a language, see its phoneme inventory.

## URL back-compat

After the phogra → glossa rename, `/phogra/…` URLs still work. The
platform serves a tiny redirect stub from
`/opt/tw_platform/apps/phogra/frontend/index.html`, which rewrites any
`/phogra/<rest>` to `/glossa/<rest>` (preserving path, query, and hash).
The stub is regenerated by `scripts/deploy.sh` on every deploy — don't
hand-edit it. When the old name has fully faded from bookmarks and
search results, the stub (and its `apps/phogra/` directory) can be
deleted.

## Data pipeline

This section is the walkthrough of how raw upstream data becomes the
JSON bundles the app renders. It's here for three audiences: contributors
who want to regenerate or extend the pipeline, AI agents modifying it,
and future-us who will want to inject **corrective edits** (see
[Roadmap](#roadmap)) at a well-defined point.

### 1,000-foot view

```
  external sources                 data-prep/ (Python)              public/data/ (JSON)          src/ (React)
  ━━━━━━━━━━━━━━━━━━━              ━━━━━━━━━━━━━━━━━━━━━            ━━━━━━━━━━━━━━━━━━━          ━━━━━━━━━━━━
  PHOIBLE 2.0 (CSV)  ──fetch──►  cache/phoible.csv ──parse──►  df ──emit──►  inventories/{iso}.json
                                                                            languages.json             ┐
                                                                            phoneme-index.json         │
                                                                                                        ├──► Zod-validated on load
  WikiPron TSVs      ──fetch──►  cache/wikipron/*.tsv ──parse──►  per-iso entries                      │    by useLanguage, useInventory,
                                                       ──emit──►  lexicons/{iso}.json                  │    useLexicon, useGraphemePhoneme
                                                                  grapheme-phoneme/{iso}.json  ◄──┐    │
                                                                  lexicons-index.json             │    │
                                                                                                   │    │
                                 reconcile.py  ────────────────────────────────────────────────── ┘    ┘
                                 (canonicalize WikiPron phonemes against PHOIBLE inventory)
```

Every source adapter module conforms to the same three-function plugin
interface:

```python
fetch(cache_dir: Path) -> Path      # download raw data (cached on disk)
parse(raw_path: Path) -> Any        # parse to an intermediate in-memory form
emit(parsed, out_dir: Path) -> None # write JSON bundles to public/data/
```

Adding a new source = drop a module in
`data-prep/glossa_data_prep/sources/` and register it in `cli.py`'s
`SOURCES` dict. `public/data/` is **gitignored**: bundles are ~80 MB and
fully derivable.

### PHOIBLE 2.0 — phoneme inventories

Canonical source: https://github.com/phoible/dev (CSV mirror of phoible.org).
License: CC BY-SA 3.0. One row per (language × inventory × phoneme), with
a column per distinctive feature.

**fetch** — pulls
`https://raw.githubusercontent.com/phoible/dev/master/data/phoible.csv`
into `data-prep/cache/phoible.csv` (skipped if already on disk).

**parse** — `pd.read_csv` → DataFrame, with a little normalization:
`Marginal` → bool, `SegmentClass` lowercased, missing `Allophones` → `""`.

**emit** — for each ISO 639-3 code, find every inventory PHOIBLE has
(English has 7: RP, American, NZ, Tyneside, …), key the largest as the
*primary* (`key = iso`), and all the others as `key = "{iso}-{invId}"`.
Writes:

| File | Contents |
|---|---|
| `public/data/languages.json` | flat list of every (iso, inventoryId), `isPrimary`, phoneme count, display name |
| `public/data/inventories/{key}.json` | per-inventory phoneme list + distinctive features per segment |
| `public/data/phoneme-index.json` | `{segment: [iso, iso, …]}` — cross-linguistic rarity index |

### WikiPron — word-level pronunciations and grapheme alignment

Canonical source: https://github.com/CUNY-CL/wikipron (Apache 2.0 code,
CC BY-SA data). Files named
`<iso3>_<script>[_<dialect>]_<broad|narrow>.tsv`, each line
`word<TAB>space separated IPA`. `broad` ≡ phonemic, `narrow` ≡ phonetic.

**fetch** — lists the remote TSV directory via the GitHub contents API,
filters to the ~15 suggested ISOs (`SUGGESTED_ISOS` in `wikipron.py`),
downloads each into `data-prep/cache/wikipron/`. Some languages are
remapped from macrolanguage sources: e.g. WikiPron has only `hbs_*`
(Serbo-Croatian) TSVs, and `_ISO_REMAPS` splits them into Latin-only for
Croatian (`hrv`) and Cyrillic-only for Serbian (`srp`).

**parse** — walks each TSV, builds `{iso: [{word, pronunciation, kind}]}`,
shuffled with a fixed seed (`0xC0FFEE`) and capped at
`MAX_ENTRIES_PER_LANG = 2000` per language.

**emit** — for each language writes:

| File | Contents |
|---|---|
| `public/data/lexicons/{iso}.json` | sampled `{word, pronunciation: [IPA], kind}` entries |
| `public/data/grapheme-phoneme/{iso}.json` | grapheme↔phoneme mapping (see below) |
| `public/data/lexicons-index.json` | per-language counts (entries, distinct phonemes, alignment pairs) |

The grapheme↔phoneme summary is built by `_build_alignment()`:

1. **Drop prosodic markers** (`ˈ ˌ ‿ . | ‖`) via `reconcile.strip_prosody`
   so stress/liaison never produce graph edges.
2. **Canonicalize phonemes** against the PHOIBLE inventory for the same
   ISO (see next section).
3. **1-to-1 align** only when `len(word) == len(segments)`. Otherwise the
   entry feeds the "example words for phoneme X" pool but contributes no
   edges. This is a deliberately dumb v1 — see
   [Roadmap → polygraphs](#roadmap).
4. **Count** pairs, keep example words per edge (shortest first, capped
   at 8) and per phoneme (capped at 5).

### Reconciliation — WikiPron × PHOIBLE

WikiPron writes broad phonemic transcriptions; PHOIBLE annotates with
combining diacritics for fine phonetic detail. Naive alignment produces
graph edges pointing at WikiPron symbols that don't appear on the IPA
chart. `reconcile.py` builds a per-language resolver that maps a
WikiPron symbol onto a PHOIBLE inventory segment using progressively
looser matching, most-specific first:

1. **Exact** — the WP phoneme is already an inventory segment.
2. **Length-strip** — `aː` → `a` if `aː` isn't in the inventory.
3. **Tie-bar-strip** — `d͡ʒ` → `dʒ` (U+0361 removed).
4. **Full normalize** — strip all combining marks + modifier letters
   (`ː ˑ ˈ ˌ ʰ ʱ ʷ ʲ ˠ ˤ`); matches `a` → `a̟`, `k` → `kʰ`.
5. **Orphan** — no match; emitted as-is into `orphanPhonemes` for
   visibility, so we can spot systematic mismatches.

### Where corrective edits will go

The pipeline is currently *write-only* — upstream data flows through,
we have no mechanism for hand-tuning prepared outputs. Known wrong-ish
outputs today include:

- Greedy 1-to-1 alignment ignores digraphs: English `sh → ʃ` shows as
  `s → ʃ` + `h → ʃ` in the graph.
- "English (New Zealand)" is the primary `eng` inventory because
  PHOIBLE's NZ inventory happens to be the biggest — but most users
  expect General American or RP.
- Several segments end up as `orphanPhonemes` because of PHOIBLE/WikiPron
  notation drift that the normalize pass doesn't catch.

Planned mechanism (not built yet): **overlays**, a small set of
hand-curated JSON files under `data-prep/overlays/` that the `emit` step
merges after the automatic pipeline runs. Shape TBD but roughly:

```json
// data-prep/overlays/eng.json
{
  "preferredInventoryId": 2175,
  "addMappings":    [{ "grapheme": "sh", "phoneme": "ʃ", "examples": ["ship"] }],
  "removeMappings": [{ "grapheme": "s",  "phoneme": "ʃ" },
                     { "grapheme": "h",  "phoneme": "ʃ" }],
  "phonemeAliases": { "ʧ": "tʃ" }
}
```

Overlays are small, hand-written, and checked in — which means
corrections are auditable, reviewable, and composable with future
pipeline changes.

## Architecture

```
data-prep/           Python (uv) — sources + orchestrator + reconcile.py
public/data/         Generated JSON (gitignored, regenerable)
src/
  schemas/           Zod schemas — SSOT for all on-disk formats, TS types inferred
  lib/               IPA utils, feature formatting, alignment (v1), data facade
  hooks/             useLanguage, useInventory, useAudio, useIpaChart
  store/             zustand slices (languages, selection, audio, ui)
  features/          lang-list, chart, phoneme-detail, graph-view
  components/        presentational primitives
  routes/            /, /lang/:iso, /lang/:iso/phoneme/:symbol, /lang/:iso/compare/:other
```

Design principles: progressive disclosure, composition over inheritance,
schema-first (every on-disk JSON validated with Zod on load), plugin-shaped
data sources.

## Known limitations

- **Multiple inventories per ISO** — PHOIBLE often has several inventories per
  language (e.g. English has ~5–7). The largest is chosen as primary;
  surfacing the choice in the UI is planned.
- **Language metadata is thin** — family, coordinates, macroarea are `null`
  until the Glottolog source adapter lands.
- **Phoneme-level audio is approximate** — Wikimedia Commons recordings are
  language-agnostic (e.g. "voiceless bilabial plosive"). For
  phoneme-in-context audio, future work can integrate Common Phone
  (6 languages, 116h aligned).
- **Alignment is greedy 1:1** — see next section.

## Roadmap

### 1. Editable prepared data (corrective overlays)

See [Where corrective edits will go](#where-corrective-edits-will-go).
Goal: make the bad outputs of the automatic pipeline fixable **without**
forking upstream data or hand-editing generated bundles. Design sketch:
overlay files in `data-prep/overlays/{iso}.json`, merged during `emit`,
so the app keeps loading the same shape of bundle.

### 2. Multi-character graphemes (digraphs, trigraphs, …)

Current model: one written character → one phoneme. This works for
languages with shallow, roughly mono-character orthographies; it fails
for French (`ai`, `eau`, `ille`, `ch`, `gn`), English (`sh`, `th`, `ch`,
`ough`), German (`sch`, `tsch`), etc. We need the **grapheme** to be a
*group of characters that maps to a single phoneme*.

Terminology note — in orthography/phonology literature this unit goes
by several names and the field hasn't fully settled:

- **grapheme** in the functional sense (what glossa uses): the smallest
  unit of writing that corresponds to a phoneme. Under this definition
  `sh`, `eau`, and `ch` are each *one* grapheme.
- **polygraph** (less common): explicitly a multi-character grapheme.
  `digraph` / `trigraph` / `tetragraph` for 2/3/4-char instances.
- **grapheme cluster** (Unicode usage): the smallest *visually*
  indivisible unit, e.g. `é` as `e` + combining acute. Narrower than the
  functional sense: a cluster is about rendering, not phonology.

Implementation direction:

- Represent each alignment entry as `{ graphemes: string[], phonemes: string[] }`
  (both arrays, same length — so a single word is a walk through the
  grapheme sequence paired with the phoneme sequence).
- Replace the current `len(word) == len(segments)` check with a proper
  per-language **segmentation** pass: given a word and its IPA, find the
  most likely split of the word into graphemes that aligns 1:1 with
  phonemes. Greedy left-to-right with a longest-match tie-break is a
  reasonable v1; EM alignment (Jiampojamarn et al.) is the principled
  answer.
- Per-language grapheme inventories (either hand-authored or mined from
  the lexicon) gate the segmenter so we only propose known graphemes.
- Overlays (item 1) let us patch individual entries.

### 3. Scripts that aren't alphabetic

Grapheme↔phoneme alignment assumes an alphabet — one character, roughly
one phoneme. That breaks for:

- **Syllabaries** — hiragana/katakana, Cherokee: one character ≈ one
  syllable (CV), so the natural unit is syllable, not phoneme. Model as
  syllable↔phoneme-cluster alignment; the grapheme tab can surface "this
  syllable splits into these phonemes" rather than character-level edges.
- **Abugidas** — Devanagari, Ethiopic, Khmer: consonant letters carry an
  inherent vowel that can be modified by diacritics. The "grapheme" is
  the akshara (consonant + any vowel mark), which still maps to one or
  two phonemes, so a modest generalization of the alphabetic pipeline
  covers it.
- **Logographic scripts** — Chinese, to some extent Japanese kanji:
  character → morpheme, not phoneme. These should show "phonetic
  readings" rather than attempt phoneme-level edges.

Strategy: add a `writingSystem` field per language (from
Glottolog/Unicode Scripts) and dispatch to different alignment
strategies. Logographic languages fall back to a read-only "here are
the phonemes and some example words" view, no mapping graph.
